# План развития настроек пользователя

## Контекст

В проекте уже добавлено JSON-поле `settings` в модель `User`. Оно служит единым хранилищем пользовательских предпочтений без необходимости в Alembic/миграциях при добавлении новых параметров.

## Реализовано (сейчас)

### 1. Настройка «не присылать пустую рассылку» (`skip_empty`)

- Поле `settings.skip_empty: bool` (default `false`).
- Команда `/skipempty` — переключатель: вкл/выкл/статус.
- Проверка в `daily_summary`: если на сегодня нет событий и `skip_empty == true`, бот молчит.

## В плане (на будущее)

### 2. Индивидуальное время рассылки

**Описание:**
Вместо жёсткого `08:08` для всех, каждый пользователь может задать своё время утреннего напоминания.

**Способ реализации (рекомендуемый: Variation B):**
- В `settings` добавить ключи `summary_hour` и `summary_minute` (default 8 и 8).
- Оставить **один** APScheduler job с `trigger='interval', minutes=1`.
- Каждую минуту job выбирает из БД пользователей, у которых `summary_hour:summary_minute` совпадает с текущим, и рассылает им `daily_summary`.

**Почему один job в минуту, а не job на каждого пользователя:**
- Нет риска утечки/размножения задач.
- Изменения настроек применяются мгновенно (не нужно пересоздавать job).
- Для SQLite один SELECT в минуту — пренебрежительно мало.

**Плюсы:**
- Пользователь выбирает удобное время (например, 7:00 или 10:30).
- Не нужно перезапускать бота при смене настроек.

**Минусы:**
- При тысячах пользователей нужен индекс по вычисляемым полям или кэширование в памяти (Redis/Memcached). Для текущего масштаба неактуально.

**Команды:**
- `/settime HH:MM` — установить время.
- `/resettime` — сбросить на дефолт (08:08).

---

### 3. Отключение рассылки на определённые дни недели

**Описание:**
Возможность выбрать дни, когда рассылка не нужна (например, выходные).

**Поля:**
`settings.active_days: list[int]` (0=пн ... 6=вс, по умолчанию все).

**Логика:**
В `daily_summary` проверяем текущий `weekday()`; если он не в `active_days` — пропускаем пользователя.

---

### 4. Настройка часового пояса

**Описание:**
Сейчас `TIMEZONE` глобальный (`Europe/Moscow`). Можно дать каждому пользователю выбрать свой TZ.

**Поля:**
`settings.timezone: str` (default `TIMEZONE` из config).

**Логика:**
В `daily_summary` использовать `ZoneInfo(user_settings.timezone)` для вычисления «сегодня».

---

### 5. Сводка на неделю вместо/в дополнение к сегодня

**Описание:**
Часть пользователей хотят не «только сегодня», а «сегодня + завтра» или «неделя».

**Поля:**
`settings.summary_mode: Literal["today", "today_tomorrow", "week"]`.

**Логика:**
В `daily_summary` формировать блок `lines` в зависимости от `summary_mode`.

---

## Архитектурные принципы для всех пунктов

1. **JSON-поле `settings`** — новый ключ = новая настройка, без миграций.
2. **Миграция на старте** — проверять и создавать колонку `settings`, если она отсутствует (уже реализовано).
3. **Команды настройки** — живут в `bot/handlers/settings.py`.
4. **Scheduler** — оставляем один job на минуту (или один `cron`, если не меняется); логика фильтрации полностью внутри `daily_summary`.
